Developer Discipline

Organization Matters

The difference between a hobby project and a professional one isn't the code // it's the structure.

Chaos Mode

  • Files named "final_v2_REAL_final.js"
  • No README // no docs // no comments
  • Passwords committed to git
  • "It works on my machine"
  • Deploy by dragging files to FTP

Organized Mode

  • Clear folder structure // named logically
  • README explains what, why, and how
  • Environment variables for secrets
  • Reproducible builds with package.json
  • Automated deployment on push
Execution

The Art of Shipping

A working demo beats a perfect plan every single time. You have to actually get things out the door.

  • 1
    Done is better than perfect.
    Ship a working v1. Iterate. The world doesn't wait for your "final touches."
  • 2
    Set a deadline and stick to it.
    Our Thursday demo tradition // every week, something ships. No exceptions.
  • 3
    Reduce scope, not quality.
    If you can't finish everything, cut features // don't cut corners on what you keep.
  • 4
    Show your work.
    Deploy it. Let people see it. Get feedback. That's how you learn and grow as a developer.

What MVP Really Means

It's not "the worst version of your product." It's the smallest version that solves the core problem.

  • Identify the ONE thing your app must do
  • Build only that feature first
  • Skip login, skip settings, skip dark mode
  • Ship it // get real user feedback
  • Then iterate based on actual data

Automation

The Automation Quest

Think of every task as a quest in a game. Some quests you grind. Some you build a bot for. Here's how to decide.

The Pencil & Paper Test

Imagine you're alone in a room with just paper and a pencil. Infinite time. No computer. How would you do this task?

  • Can you write the steps as a checklist?
  • Does every run follow the same steps?
  • Does it need a human to approve or judge?
  • Could you teach a stranger to do it?

If yes to all // it's deterministic. A machine can do it.

// Shoutout Teja for this framework

The Quest Slot System

Think of your tasks as inventory slots in a game. Each slot costs energy to maintain.

  • 1-5 times // Just do it. Manual is fine.
  • 5-25 times // Write a checklist or snippet.
  • 25+ times // Automate it. Free up the slot.
  • Needs approval? // Never fully automate.

Every automation frees a slot for creative work.

"Automation isn't about being lazy // it's about freeing up quest slots for the boss fights that actually matter."

MVP is the most misunderstood concept in tech.

People think MVP means "the worst version." It actually means "the smallest version that proves the idea."

"Your MVP should answer ONE question: does anyone care? If nobody uses the simplest version, adding features won't help."

Walk through the Last Thursday evolution:

  • Static HTML: Just shows projects. No login, no database. Proves the concept works.
  • JavaScript data: Projects stored in JS files. Still no backend. Proves we need structured data.
  • Full-stack: Database, auth, CRUD. Now multiple people can add projects. Proves collaboration works.
  • Roles + Media: Admin panel, YouTube, GitHub links. Now it's a real platform.

Each iteration answered a specific question before moving to the next. That's MVP done right.

Anti-pattern: Don't build a login system before you know if anyone wants your product. Don't add payments before you have users. Don't optimize before you have traffic.

This slide reframes automation as a game design problem. Two frameworks from class.

Framework 1 // The Pencil & Paper Test (credit: Teja)

Teja described this perfectly: before you automate anything, imagine you're alone in a room with just paper and a pencil and infinite time. Write out every step. If you can produce a deterministic checklist // same input, same steps, same output every time // then a machine can do it. If at any point you write "use your judgment" or "it depends" // that's a human task.

"If you can't explain it to a piece of paper, you can't explain it to a computer. Automation only works on deterministic tasks // tasks where the steps never change."

Walk through examples with the class:

  • Deterministic: "Resize every image in this folder to 800px wide" // same steps every time, no judgment. Automate it.
  • Deterministic: "Run tests and deploy if they pass" // clear pass/fail, no ambiguity. Automate it.
  • NOT deterministic: "Review this pull request" // requires understanding context, intent, quality. Human task.
  • NOT deterministic: "Write a good commit message" // requires understanding WHY the change was made. Human task.

Framework 2 // The Quest Slot System

Think of your brain like an RPG inventory. You only have so many quest slots. Every repetitive task you're doing manually is taking up a slot that could be used for creative work // the boss fights.

The tier system:
1-5 times: Just do it by hand. The automation would take longer than all 5 runs combined.
5-25 times: Write a checklist, a code snippet, or a shell alias. Semi-automated. You still run it, but it's faster.
25+ times: Full automation. Script it, schedule it, CI/CD it. This task should never require your attention again.
Needs human approval? Never fully automate. Build a tool that does 90% of the work and presents options to a human. The human presses the button.
"Every task you automate frees up a quest slot. If you're spending all your slots on grinding // deploying manually, formatting code, copying files // you have no slots left for the creative work that actually matters. Automate the grind so you can fight the bosses."

Real examples from our class:

  • Deployment // we do this 25+ times per project. Vercel automates it. Quest slot freed.
  • Code formatting // Prettier runs on save. You never think about tabs vs spaces again. Quest slot freed.
  • Database migration // we did this once today. Just ran the SQL. No automation needed.
  • Approving projects // requires a moderator's judgment. Semi-automated: the system shows drafts, human clicks "Approve."

The bottom quote ties it together: automation isn't about being lazy. Lazy is doing the same manual task for the 50th time. Automation is being strategic about where you spend your limited energy.

CI/CD is the backbone of modern software development.

Walk through the pipeline diagram step by step:

  • Write Code: On your laptop, in your editor. This is where creativity happens.
  • Git Push: You push to GitHub. This triggers everything else.
  • Auto Build: Vercel (or GitHub Actions, or Jenkins, etc.) pulls your code and runs the build command.
  • Run Tests: Automated tests catch bugs before they reach production. If tests fail, deployment stops.
  • Deploy: If everything passes, your code goes live. Users see the new version.
Our setup: We use Vercel's built-in CI/CD. Push to main = auto deploy. No servers, no SSH, no FTP. It's free for small projects. This is the modern standard.

Mention alternatives: GitHub Actions (free, flexible), Jenkins (self-hosted), GitLab CI, CircleCI. The concept is the same everywhere.

"The old way: FTP into a server, overwrite files, pray nothing breaks. The new way: git push, get a coffee, it's live."
