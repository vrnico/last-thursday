The Stack

Four Pieces, That's It

Next.js

React framework. Gives us pages, API routes, and server-side rendering. One command to create a project.

Auth.js

Handles GitHub OAuth. Three files and you have login/logout. No passwords, no sessions to manage manually.

Vercel Postgres

SQL database in the cloud. Free tier. Create tables, store projects, query data. Real SQL, not a toy.

Vercel

Deploy with git push. Env vars stay on the server. Your secrets never hit the browser.

BREAKPOINT 5

Roles & Moderation

  • user // can create projects, edit/delete their own. New projects start as draft
  • moderator // can approve drafts to published, edit/delete any project
  • The flow: user creates a project → it's a draft → mod reviews it → approves or sends feedback → it goes live on the site
  • One line of code separates these roles. It's just a column in the users table. No magic. No special framework.
lib/permissions.ts export function canEditProject(user, project) { if (user.role === "moderator") return true return user.id === project.author_id } export function canApprove(user) { return user.role === "moderator" } export function canViewProject(user, project) { if (project.status === "published") return true if (!user) return false return user.role === "moderator" || user.id === project.author_id }
BREAKPOINT 6

Deploy to Vercel

  • Push to GitHub // Vercel watches the repo. Every push triggers a new deploy. Automatic.
  • Environment variables // GITHUB_ID, GITHUB_SECRET, POSTGRES_URL go in the Vercel dashboard. They never touch your code. They never go in git.
  • Why this matters: keys in code = keys in public. Vercel's env vars are encrypted, server-side only. Even if someone reads your GitHub repo, they can't get your secrets.
  • Custom domain? One click in Vercel. Point a domain at it and you're live. Or use the free .vercel.app URL.

Slide 3: The Stack

~5 minutes
Keep it simple: "Four tools. That's the entire stack. Next.js gives us a React app with built-in API routes // meaning the frontend and backend live in the same project. Auth.js handles the GitHub login. Vercel Postgres is our database. And Vercel is where we deploy."
Why Next.js: "You could build this with plain HTML/CSS/JS. We did that with the static version. But the moment you need a server // for auth, for database queries, for keeping secrets // you need something that can run code on a server. Next.js gives you that without having to set up a separate backend. One project, one deploy."
Why Auth.js: "Building a login system from scratch is a bad idea. Not because it's hard // it's not that hard // but because there are a million ways to get it wrong. Session hijacking, CSRF, token expiry. Auth.js handles all of that. We configure it, we don't build it. That's a smart engineering decision, not a lazy one."
Why Vercel Postgres: "We need somewhere to store projects. Could use a JSON file, could use Firebase, could use MongoDB. We're using Postgres because it's real SQL // the same language used by every bank, hospital, and government database on earth. Learning SQL is never wasted time."

Analogy

"Think of it like building a restaurant. Next.js is the building. Auth.js is the door lock // it lets the right people in. Postgres is the kitchen // where the food (data) is prepared. Vercel is the neighborhood // where the building exists so people can find it."

Breakpoint 2: GitHub OAuth

~10 minutes
Walk through the flow diagram: "OAuth is a dance between three parties. The user, GitHub, and our server. Step 1: user clicks 'Sign in.' Step 2: they get sent to GitHub's website. GitHub asks 'Do you want to give Last Thursday access to your profile?' Step 3: user says yes, GitHub sends a code back to our server. Step 4: our server takes that code and exchanges it for a real access token. Step 5: we have a session. The user is logged in."
Why this is better than passwords: "We never see the user's password. We never store a password. We never have to worry about someone hacking our database and getting passwords. GitHub handles all the authentication. We just get the result: 'This person is real, here's their username and avatar.'"
Show the code: Open the actual route.ts file. "Look how little code this is. We import NextAuth, import the GitHub provider, configure it with our client ID and secret, and export the handler. That's the entire auth system. Auth.js does the rest. The [...nextauth] folder name is a catch-all route // it handles /api/auth/signin, /api/auth/signout, /api/auth/callback, all from this one file."

Show the GitHub OAuth app setup

Open GitHub → Settings → Developer settings → OAuth Apps. "This is where you create the OAuth app. You give it a name, a homepage URL, and a callback URL. GitHub gives you a Client ID and Client Secret. Those go in your .env.local. That's the handshake // GitHub knows about your app, your app knows about GitHub."
Key concept // environment variables: "Notice the code says process.env.GITHUB_ID. It doesn't say the actual ID. The value comes from the environment // from .env.local locally, and from Vercel's dashboard in production. If someone reads our code on GitHub, they see the variable name but never the value. This is how every professional app handles secrets."

Breakpoint 3: The Database

~8 minutes
What is a database: "A database is just organized storage. Think of it like a spreadsheet with rules. Each table is a sheet. Each row is a record. Each column is a field. The difference between a spreadsheet and a database is that a database enforces rules // this column must be unique, that column can't be empty, this row must reference a row in another table."
Walk through the schema: "We have two tables. users stores everyone who logs in via GitHub. ID, their GitHub ID, username, avatar URL, and a role column that defaults to 'user'. projects stores the project entries. Title, description, week number, who created it (linked to users via author_id), and a status field that's either 'draft' or 'published'."
The foreign key: "See author_id in the projects table? That's a foreign key. It references the id column in the users table. This is how we know who created each project. If I look up project #5, I can follow the author_id to find the user who made it. This is relational data // two tables linked by a shared value. That's why it's called a relational database."

Show the Vercel Postgres dashboard

Open the Vercel dashboard and navigate to the Postgres section. "This is where the database lives. You can run SQL queries right here in the browser. Let me run SELECT * FROM users; and show you what's in there." Run it live. Show the data. "That's real data. Real rows. Every time someone logs in with GitHub, a row appears here."
The draft/published pattern: "Notice the status column. New projects start as 'draft'. They don't show up on the public site until a moderator changes them to 'published'. This is a common pattern // it's how WordPress works, how news sites work, how any CMS works. You've been using this pattern every time you read an article online. Now you know how it works underneath."

Walk through the POST route: "Let's read this code line by line. First: getServerSession(). This checks if the person making the request is logged in. If not, we return a 401 error // 'Not logged in.' This is authorization. You must be authenticated to create a project."
The SQL: "Next: sql\`INSERT INTO projects...\`. This is a SQL query that creates a new row in the projects table. Breakpoint 5: Roles & Moderation

~7 minutes
Keep it grounded: "Roles are just a string in a database column. That's it. There's no magic. The user table has a role column. It defaults to 'user'. If I go into the database and change someone's role to 'moderator', they can now approve projects and edit anyone's work. That's how access control works in every app."
Walk through the permission functions: "Look at these two functions. canEditProject returns true if you're a moderator OR if you're the author of the project. canApprove returns true only if you're a moderator. These are called every time someone tries to edit or approve something. If the check fails, the API returns a 403 // 'Forbidden.' Simple if-statements protecting your data."
The approval flow: "A user creates a project. It starts as 'draft'. It doesn't show up on the public site. A moderator logs in, sees the pending drafts in their dashboard, reviews them, and clicks 'Approve'. That changes the status from 'draft' to 'published'. Now it shows up on the site for everyone. If something's wrong, the mod can edit it or delete it."

Discussion

"Who should be a moderator for our site? How do you decide who gets that power? This is a real governance question. Every online community has it. Discord has mods. Reddit has mods. Subreddits have their own mods. Wikipedia has editors. How should we handle it?"
The bigger lesson: "Authorization is one of the most important concepts in software. Authentication is 'who are you?' Authorization is 'what are you allowed to do?' Every data breach you've ever heard of was an authorization failure. Someone accessed data they weren't supposed to. Understanding this makes you a better engineer and a more critical user of technology."

Breakpoint 6: Deploy to Vercel

~5 minutes
The deploy process: "Deploying to Vercel is almost anticlimactic. You connect your GitHub repo. You set your environment variables in the Vercel dashboard. You push your code. Vercel automatically builds it, deploys it, and gives you a URL. Done. No server to configure. No Docker. No AWS console. Just push and it's live."
Environment variables in Vercel: Show the Vercel dashboard. "Same four env vars we had in .env.local. But now they live here, encrypted, on Vercel's servers. When our code runs, process.env.GITHUB_ID pulls from these values. The code is the same whether it's running locally or in production. Only the environment changes."
Security lesson: "This is why environment variables matter. Your code is public on GitHub. Anyone can read it. But the secrets // your GitHub OAuth credentials, your database connection string // those are only on Vercel's servers and your local machine. If someone forks your repo, they get the code but not the keys. They can't log in, they can't access your database. Separation of code and secrets. That's security 101."
